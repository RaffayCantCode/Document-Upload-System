1. How It Was Made

This project was built for a university admissions portal where applicants need to upload supporting documents — transcripts, CNIC, and passport photos — as part of their application. The requirements were straightforward: drag-and-drop uploads, file type/size validation, status tracking for administrators, and file preview before and after upload.

The development process went through several iterations:

Phase 1 — Core Backend

I started with the Express backend using a layered architecture: routes → controllers → services → repository. The repository initially used sql.js, a pure JavaScript port of SQLite compiled via Emscripten. This was chosen because it doesn't require any native compilation tools — no Python, no C++ compiler — just npm install and it works on any machine. Files are stored as BLOBs directly in the database rather than on the file system, which simplifies deployment (no shared file storage needed).

Phase 2 — Frontend Widgets

The React frontend was built as a set of reusable widgets (DocumentUpload, DocumentList) with scoped CSS (all classes prefixed doc-*). This was intentional: the widgets can be dropped into any existing university portal without breaking existing styles. The dark gradient theme with glassmorphism cards was a design choice to make the demo look polished.

Phase 3 — Pluggable Database

Originally the backend was hard-coded to use sql.js. I realized that real universities already have PostgreSQL databases, so I restructured the entire backend to use dependency injection: the repository is a factory function, the service accepts it as a parameter, and the module factory wires everything together. Now anyone can swap in PostgreSQL, MySQL, MongoDB, or any other database by passing a custom repository object.

Phase 4 — PostgreSQL & Fixes

I connected the system to a live Supabase PostgreSQL instance. This uncovered several bugs: the service and controller methods were synchronous (worked with sql.js's sync API but broke with PostgreSQL's async pg library), and the .env file was never actually loaded (no dotenv). Both were fixed. A CSS overlay bug was also found — the dropzone's ::before pseudo-element was eating click events on the Preview and Remove buttons because it lacked pointer-events: none.

2. Architecture Deep Dive

The system follows a strict layered architecture where each layer only talks to the one directly below it. This makes the system testable, maintainable, and swappable.

┌─────────────┐     ┌──────────────────────────────────────────────┐
│  Frontend   │     │  React 19 + Vite                             │
│  (port 3000) │     │  DocumentUpload, DocumentList, useDocuments  │
└──────┬──────┘     └──────────────────────────────────────────────┘
       │  HTTP (fetch)
       ▼
┌─────────────┐     ┌──────────────────────────────────────────────┐
│  Backend    │     │  Express.js                                  │
│  (port 5000) │     │                                              │
│             │     │  Router → Controller → Service → Repository  │
└──────┬──────┘     └──────────────────────────────────────────────┘
       │  SQL queries
       ▼
┌─────────────┐     ┌──────────────────────────────────────────────┐
│  Database   │     │  SQLite (sql.js) or PostgreSQL (pg)          │
│             │     │  or any custom adapter                       │
└─────────────┘     └──────────────────────────────────────────────┘

Key design rule: The service layer never touches Express or HTTP. It only calls repository methods and returns plain objects. The controller is the only layer that interacts with req and res. This means you can test the service logic without a running server.

3. Backend Layers Explained

Config (backend/config/index.js)

Reads environment variables and provides a single config object used by every other layer. DB_TYPE determines which database adapter to use. maxFileSizeMB controls the upload limit. allowedMimeTypes defines which file formats are accepted (PDF, PNG, JPG).

Database Init (backend/config/db.js)

Initializes the sql.js database engine, creates the documents table if it doesn't exist, and provides getDb()/saveDatabase() helpers. The saveDatabase() function is critical — sql.js is an in-memory database, so every write must explicitly export the buffer to disk or data is lost on restart.

Middleware (backend/middleware/upload.js)

Uses Multer with memoryStorage to handle file uploads. The file is stored in req.file.buffer as a byte array. Memory storage was chosen over disk storage because the body fields (applicant_id) weren't available in Multer's destination callback — they're parsed after the file stream starts.

Routes (backend/routes/documents.js)

Maps HTTP methods and paths to controller functions. Six endpoints: upload, list, get metadata, download, delete, update status. No logic here, just wiring.

Controllers (backend/controllers/documentController.js)

Extracts data from the request (req.params, req.query, req.body, req.file), calls the service, and sends the response. Uses an asyncHandler wrapper to catch promise rejections and forward them to the error middleware.

Services (backend/services/documentService.js)

Contains all business logic: validates file existence, checks document type against allowed list, generates UUIDs for document IDs, ensures status transitions are valid. Every method is async and awaits all repository calls — this is essential for PostgreSQL compatibility.

Repository (backend/db/documentRepository.js)

The database adapter. The default implementation uses sql.js with raw SQL queries. Returns plain objects. The PostgreSQL equivalent (backend/db/postgresRepository.js) uses the pg package with connection pooling. A template (backend/db/templateRepository.js) is provided for custom databases.

Module Factory (backend/index.js)

The entry point for integrators. createModule({ repository }) wires together the service, controller, and routes. If no repository is provided, it auto-selects based on config.dbType. This is the "pluggable" part — swap the database by passing a different repository object.

4. Frontend Components Explained

DocumentUpload (frontend/src/components/DocumentUpload.jsx)

A form widget with three parts:

  • Document type selector — dropdown for transcript, CNIC, or photo
  • Dropzone — supports drag-and-drop and click-to-browse. When a file is selected, it shows the file name, size, and two buttons: Preview (opens a modal with URL.createObjectURL) and Remove (clears the selection)
  • Submit button — uploads the file via fetch() to the backend, shows a shimmer progress bar during upload

DocumentList (frontend/src/components/DocumentList.jsx)

A table widget that displays uploaded documents with columns: type (badge), file name (clickable for preview), size, upload date, status badge (pending/verified/rejected), and actions (Preview, Download, Delete). Includes a filter dropdown and loading/empty states.

PreUploadPreview (frontend/src/components/DocumentUpload.jsx)

A modal that shows a preview of the selected file before uploading. Images display inline using URL.createObjectURL(). PDFs show a message that preview is only available after upload (browsers can't reliably preview local PDF blob URLs).

PostUploadPreview (frontend/src/components/DocumentList.jsx)

A modal that shows the file fetched from the database. Images display inline via the download endpoint URL. PDFs offer an "Open in new tab" link. A Download button is always available.

useDocuments Hook (frontend/src/hooks/useDocuments.js)

A React hook that encapsulates document state management: fetching the list when applicantId changes, deleting documents with optimistic UI updates, and exposing loading/error states.

API Client (frontend/src/api.js)

A factory function createApiClient(baseUrl) that returns an object with methods: upload, list, get, remove, updateStatus, and getDownloadUrl. Each method handles JSON parsing and error checking.

State Persistence (frontend/src/App.jsx)

The applicant ID is saved to sessionStorage whenever it changes. If the page reloads (Vite HMR, browser refresh, tab switch), the ID is restored and documents are fetched again automatically.

5. Database Design & Why BLOBs

The documents table schema:

Column          | Type      | Notes
----------------|-----------|-----------------------------------
id              | TEXT (PK) | UUID generated by the service
applicant_id    | TEXT      | e.g. "APP-2024-001"
document_type   | TEXT      | transcript | cnic | photo
file_name       | TEXT      | Original filename
file_data       | BLOB      | The actual file bytes
file_size       | INTEGER   | Bytes
mime_type       | TEXT      | application/pdf | image/png | image/jpeg
uploaded_at     | TIMESTAMP | Auto-set
status          | TEXT      | pending | verified | rejected (default: pending)

Why BLOBs instead of file system? In a university admissions system with multiple backend servers behind a load balancer, storing files on the local disk doesn't work — the file might be on server A but the next request goes to server B. A shared NAS adds complexity. BLOBs in the database solve this: any server can serve any file, backups include everything, and there's no file system permission issues. The trade-off is database size, but for documents (PDFs, images at ~500KB each) it's negligible for any modern database.

6. Pluggable Database System

The module accepts any object that implements six methods:

{
  insertDocument(id, applicantId, docType, fileName, fileData, fileSize, mimeType),
  findDocuments(applicantId),
  findDocumentById(id),
  findDocumentFileById(id),
  deleteDocumentById(id),
  updateDocumentStatus(id, status)
}

Three adapters are provided out of the box:

SQLite (backend/db/documentRepository.js)

Uses sql.js — a pure JavaScript SQLite compiled via Emscripten. Synchronous API, zero dependencies, works everywhere. Data is stored in a single database.sqlite file. Every write must call saveDatabase() to persist the in-memory database to disk.

PostgreSQL (backend/db/postgresRepository.js)

Uses the pg package with connection pooling. All methods are async. Switch by setting DB_TYPE=postgres and PG_CONNECTION_STRING in .env. The table is auto-created via ensureTable().

Template (backend/db/templateRepository.js)

Empty stubs with JSDoc comments. Copy this file and implement each method for MongoDB, MySQL, Firestore, or any other database.

7. How to Run

Both servers must run simultaneously. The frontend on port 3000 sends API calls to the backend on port 5000.

Option A — Double-click start.bat

Opens two terminals automatically: one for the backend, one for the frontend.

Option B — Manual

  1. cd backend && npm install && npm start → Backend on http://localhost:5000
  2. cd frontend && npm install && npm run dev → Frontend on http://localhost:3000
  3. Open http://localhost:3000, enter an Applicant ID, select a file, upload

Environment Variables (backend/.env)

PORT=5000
MAX_FILE_SIZE_MB=1
DB_TYPE=postgres
PG_CONNECTION_STRING=postgresql://user:pass@host:5432/db

9. How to Integrate Into Your Site

Backend

const docUpload = require('./backend');
await docUpload.initDatabase();

// Default: SQLite
app.use('/api/documents', docUpload.createModule());

// Or PostgreSQL via env:
// DB_TYPE=postgres PG_CONNECTION_STRING=...

// Or custom database:
app.use('/api/documents', docUpload.createModule({
  repository: myCustomRepo
}));

app.use(docUpload.errorHandler);

Frontend

import { DocumentUpload, DocumentList, createApiClient, useDocuments } from './frontend/src/index';
import './frontend/src/App.css';

const api = createApiClient('/api/documents');

function MyPage() {
  const [applicantId, setApplicantId] = useState('');
  const { documents, deleteDocument, refresh } = useDocuments(api, applicantId);

  return (
    <div>
      <DocumentUpload apiClient={api} applicantId={applicantId} onUploadSuccess={refresh} />
      <DocumentList documents={documents} onDelete={deleteDocument} getDownloadUrl={api.getDownloadUrl} />
    </div>
  );
}

10. Challenges Faced

1. BLOB Storage with sql.js — Binary Data Quirks

sql.js is a JavaScript port of SQLite compiled via Emscripten. Storing files as BLOBs worked for INSERT, but retrieving them was tricky. The standard stmt.getAsObject() returned binary data as a typed array incompatible with Express's res.send(). I had to use stmt.get() and stmt.getColumnNames() manually and wrap the result in Buffer.from(). One wrong type conversion and the downloaded file would be corrupt despite the correct byte count.

2. Pluggable Database without a DI Framework

Restructuring from a monolithic sql.js import to a factory pattern was the hardest architectural decision. The repository, service, and controller all needed to be created via factory functions that accept their dependencies as parameters. Getting the wiring order right in index.js without a dependency injection framework required careful sequencing. The result is clean but was painful to refactor.

3. sql.js Persistence — Manual Save on Every Write

sql.js operates entirely in memory. The database file is only updated when you explicitly call db.export() and write the buffer to disk. If the server crashes between a write and a save, that data is lost forever. Every repository method must call saveDatabase() after execution. This is a fundamental constraint of WASM-compiled databases.

4. Multer Body Field Timing

Multer's diskStorage destination callback fires during file streaming, before non-file form fields are parsed. So req.body.applicant_id was undefined when trying to create the folder path. Switching to memoryStorage and writing the file after Multer finishes solved it, but required extra code to construct the path manually.

5. Two Preview Flows

Pre-upload preview uses URL.createObjectURL(file) from the local File object. Post-upload preview fetches the file from the database via the download endpoint. Both needed the same modal component but with different data sources. Managing revokeObjectURL to prevent memory leaks while letting server URLs cache required careful state management. PDFs added another complication — browsers can't reliably preview local PDF blob URLs.

6. Dotenv Was Never Loaded

The .env file existed but dotenv was never installed or called. process.env.DB_TYPE was always undefined, defaulting to sqlite. The entire PostgreSQL adapter was written but never actually used during testing. All uploads went to the local SQLite file while I was debugging PostgreSQL issues on Supabase. Fixed by installing dotenv and adding require('dotenv').config() to server.js.

7. CSS Overlay Eating Button Clicks

The dropzone's ::before pseudo-element had position: absolute; inset: 0 but no pointer-events: none. It sat as an invisible overlay on top of the Preview and Remove buttons, intercepting all click events. The button onClick handlers never fired — the ::before caught the event and bubbled it to the dropzone, which did nothing because the file was already selected. A one-line CSS fix (pointer-events: none) solved it.
