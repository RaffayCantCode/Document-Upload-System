Document Upload System

University Admissions Portal — SafeX Solutions
Stack: React 19 + Express.js + SQLite/PostgreSQL (pluggable database)

1. What It Does

University applicants upload transcripts, CNIC, and photos for admission. Files are validated (type + size) and stored as BLOBs directly in the database — no file system needed. Administrators can review and mark documents as verified or rejected.

2. How to Run

Both servers must run simultaneously:

  1. cd backend && npm install && npm start → port 5000
  2. cd frontend && npm install && npm run dev → port 3000
  3. Open http://localhost:3000, enter an Applicant ID, upload a file

Or double-click start.bat to open both terminals automatically.

3. Architecture

The backend uses a strict layered architecture:

Router → Controller → Service → Repository → Database

Router — maps URLs to controller functions. No logic.
Controller — extracts data from req/res, calls service, sends response.
Service — business logic: validates file type, size, document type; generates UUIDs. Never touches Express.
Repository — database queries. swappable — SQLite, PostgreSQL, or any custom adapter.
Module Factory (index.js) — wires everything together. createModule({ repository }) accepts any database adapter.

The frontend has three reusable widgets:

  • DocumentUpload — dropzone with drag-and-drop, type selector, pre-upload preview, remove button
  • DocumentList — table with status badges, filter, preview modal, download, delete
  • useDocuments — React hook managing document CRUD state

All CSS is prefixed with doc-* to avoid conflicts when embedded in existing portals.

5. Database & Pluggable Adapters

The documents table stores: id, applicant_id, document_type, file_name, file_data (BLOB), file_size, mime_type, uploaded_at, status. Files stored as BLOBs so any backend server can serve any file — no shared file system needed.

Three adapters provided:

  • SQLite (default via sql.js) — zero config, synchronous API, in-memory with manual saveDatabase() persistence
  • PostgreSQL (via pg) — connection pooling, async, auto-creates table
  • Template — empty stubs; copy and implement for MongoDB, MySQL, etc.

Switch with DB_TYPE=postgres + PG_CONNECTION_STRING in .env, or pass a custom repo: createModule({ repository: myRepo }).

6. Integration

Backend (3 lines):

const docUpload = require('./backend');
await docUpload.initDatabase();
app.use('/api/documents', docUpload.createModule());
app.use(docUpload.errorHandler);

Frontend:

import { DocumentUpload, DocumentList, createApiClient, useDocuments } from './frontend/src';
const api = createApiClient('/api/documents');
<DocumentUpload apiClient={api} applicantId={id} onUploadSuccess={refresh} />
<DocumentList documents={docs} onDelete={deleteDocument} getDownloadUrl={api.getDownloadUrl} />

8. Challenges

  1. sql.js BLOBs — retrieving binary data needed manual stmt.get() + Buffer.from() instead of getAsObject(). One wrong type conversion corrupts the download.
  2. Pluggable DB without DI — inverting the dependency chain so any database can be injected required factory functions for repo, service, and controller with careful wiring order.
  3. sql.js persistence — in-memory database means every write must call saveDatabase() or data is lost on crash.
  4. Multer timing — req.body undefined in diskStorage destination callback. Switched to memoryStorage + manual write.
  5. CSS overlay — ::before pseudo-element was eating button clicks; fixed with pointer-events: none.
  6. Dotenv missing — .env was never loaded, so DB_TYPE always defaulted to sqlite. Fixed by installing dotenv.
  7. Async vs sync — service/controller methods were synchronous (worked with sql.js but broke PostgreSQL's async pg). Made all methods async/await.
